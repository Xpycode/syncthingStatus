## Issues Found

### High

- [ ] **30+ bare `print()` calls in production paths** — directly undermines the new diagnostic export feature.
  - **Why this matters in v1.6.0 specifically**: `DiagnosticLogger.export()` filters by `subsystem == "com.lucesumbrarum.syncthingStatus"`. `print()` writes to stdout, which is **not part of that subsystem**, so every error currently logged via `print()` will be **invisible in user-attached diagnostic logs** — exactly the failure mode the feature was added to fix.
  - **Worst offender**: `App.swift:440` — `print("📊 Popover sizing: …")` fires on *every popover open*. Pure noise in production, and as an emoji-prefixed debug breadcrumb it should not have shipped at all.
  - **Sites** (full list):
    - `App.swift:311` notification permission error
    - `App.swift:440` popover sizing trace
    - `Client.swift:376` API key length (this one is correctly behind `#if DEBUG`)
    - `Client.swift:390, 414, 439, 462, 499` HTTP failure logs (5 sites)
    - `Client.swift:539, 577, 609, 1027, 1053, 1072, 1091, 1110, 1131` error/notification logs (9 sites)
    - `Client.swift:1172` "Refresh already in progress" trace
    - `Client.swift:1214, 1225, 1260, 1270, 1280, 1306, 1313, 1338` device/folder/config errors (8 sites)
    - `LaunchAtLoginHelper.swift:24`
    - `SyncthingSettings.swift:329, 334, 399, 434, 451, 470, 475, 505` keychain + migration logs (8 sites)
  - **Fix**: convert each to `Logger(subsystem:category:)` calls at the appropriate level (`.error` for failures, `.notice` for migrations, `.info` for progress). The infrastructure is already there — `bookmarksLog`, `stuckDeletesLog`, `appLifecycleLog` are existing patterns to copy.
  - **Effort**: mechanical; ~30 one-line replacements + adding ~5 new `Logger` instances scoped by file. Maybe 30–45 minutes.

### Medium

- [ ] **Entitlements file is empty (`<dict/>`)** — sandbox is enabled via `ENABLE_APP_SANDBOX = YES` in `project.pbxproj`, not via the file Xcode signs against.
  - This *works* (Xcode synthesizes the entitlement at build time), but it's fragile: any tooling that reads `syncthingStatus.entitlements` directly (linters, codesign re-signing scripts, CI inspection) will conclude the app is unsandboxed.
  - **Verify before release**: `codesign -d --entitlements - /path/to/syncthingStatus.app` should show `com.apple.security.app-sandbox = true` baked in. If it does, the build is fine — but consider committing the explicit entitlement to the file so it's source-of-truth, not Xcode-tab-of-truth.

- [ ] **`try? settings.updateConfigBookmark(with: url)` (Client.swift:282)** — security-scoped bookmark for `config.xml` silently swallowed on save failure. If this fails, the next launch can't read `config.xml` and the user gets re-prompted with no log trail to diagnose. Convert to `do/catch` and log via OSLog.

### Low

- [ ] **`requestNotificationPermissions` silently swallows denial** (App.swift:308) — only logs via `print()`. Acceptable if the user can intuit "notifications didn't appear → I denied them," but a Settings-page status indicator ("Notifications: Denied — open System Settings") would close the loop. Defer to v1.6.x.

- [ ] **Cleanup ⌘A select-all not live-tested** — already noted in PROJECT_STATE; visual layout verified, real interaction deferred to next stuck-delete repro.

### Out-of-scope but noted

- [ ] **Files >500 lines**: `Views.swift` 2259, `Client.swift` 2093, `App.swift` 972, `SyncthingSettings.swift` 510. Checklist flags these but splitting is a separate refactor — not a v1.6.0 release blocker. Worth scheduling as a v1.7 cleanup pass; `Client.swift` is the strongest candidate (the `StuckDeletesController` could move to its own file, ~400 LOC of self-contained code).

## Summary

**~22 of ~25 checklist items pass.** Zero Critical. One High (the `print()` pollution) is a release blocker by checklist rules, but more importantly it materially weakens the new diagnostic feature — without converting them, when a v1.6.0 user reports a bug and exports the diagnostic log, the actual error trail will be missing.

**Recommendation: do the print → Logger conversion before cutting the Release build.** The other findings are deferrable.

3. Then proceed to release notes + notarization.
