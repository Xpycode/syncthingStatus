## Decisions

- **`db/need` instead of `db/remoteneed`** for candidate listing. The design doc proposed `db/remoteneed` because the live diagnostic was run from the sender; on the receiver (where cleanup happens) `db/need` is the correct vantage point. One required parameter instead of two; one call per folder instead of one per (folder, peer).
- **Phase 0 ships standalone as the popover gap fix.** Splitting it from the destructive cleanup means the user's headline complaint ("popover doesn't match WebUI") could ship as a v1.5.6 patch independent of the heavier cleanup feature. We rolled them into v1.6.0 anyway, but the architectural separation kept Phase 0 reviewable in isolation.
- **Dedicated `NSWindow`, not a popover sheet, for cleanup.** NSPopover-attached sheets have well-known z-order bugs; popover dismissal on app deactivation would close the cleanup window unexpectedly. Mirrors `MainWindowController` pattern.
- **Per-folder window dedup with focus-existing semantics.** Repeated "Resolve…" clicks on the same folder focus the existing window rather than spawning duplicates. Multi-folder still allowed.
- **Selection state lives in the View, not the controller.** `@State Set<String>` with `.onChange(of: controller.candidates)` intersection — stale IDs evict in one line. Lifting state to the right layer simplified deletion.
- **`nonisolated static validatePath`.** Pure validator on the type, not the instance, so the detached deletion task can call it without bouncing back to the main actor.
- **`FileManager()` instances inside detached tasks**, not `FileManager.default` — Apple's documented thread-safety guidance.
- **`CocoaError` typed catches over NSError integer codes.** `.fileReadNoPermission` / `.fileWriteNoPermission` / `.fileNoSuchFile` / `.fileReadNoSuchFile` — type-safe, self-documenting.
- **ENOENT counts as success.** Concurrent windows or Finder cleanups beating us to the punch shouldn't produce spurious failures.
- **FDA gate only on real permission denial.** Catch-all error handlers in pre-flight code paths produce false positives that dead-end the user. Always classify error types and reserve the destructive UI for the case it's actually meant for.
- **Loosened `makeRequest<T: Codable>` to `<T: Decodable>`.** Response types only need `Decodable`; the `Codable` constraint forced downstream types to fake an `Encodable` conformance they never used.
- **Closure-based dismiss + FDA-opener** instead of `weak var window: NSWindow?` on the controller — kept `Client.swift` framework-agnostic (no `import AppKit`).

### Iteration 2 — flapping-bug latch fix (build 160 → 161)

OSLog showed `Stuck deletes detected` / `cleared` ping-ponging every ~60 s. Root cause in `updateStuckDeletesSignal()`: the entry fingerprint (`state == idle && needDeletes > 0 && needFiles == 0 && needBytes == 0`) was *also* the persistence fingerprint. Any periodic-rescan transition through `scanning` wiped `firstSeenStuckAt`, dropped the folder from `newCounts`, and restarted the 30-s debounce on the next idle window. The popover's "Resolve…" button was only visible ~50 % of the time.

Plus a logging bug: `lastLoggedStuckState[id] = false` never removed the key, so the `cleared` log re-fired on every poll where the key matched the where-clause.

**Fix:**
- Two-stage state machine. *Entry* still requires `state == idle` and the 30-s debounce. *Persist* is state-agnostic — once published, the folder stays in `stuckDeleteCounts` as long as `needDeletes > 0 && needFiles == 0 && needBytes == 0`. Real sync work (`needFiles > 0` / `needBytes > 0`) or `needDeletes` resolving to 0 unlatches.
- "cleared" log gated on `wasDetected == true` (the value, not the presence of the key).

### Iteration 3 — FDA gate replaced with security-scoped bookmark (build 161)

Even after granting FDA in System Settings, the gate persisted. Diagnosis on the live system:
- App is sandboxed (`com.apple.security.app-sandbox = true` baked into the build via project signing capabilities, despite the project's `.entitlements` file being empty — Xcode's "Signing & Capabilities" tab is the source of truth here).
- The probe's `contentsOfDirectory(atPath:)` returns `.fileReadNoPermission` because `/Users/sim/ProgrammingProjects` is outside the sandbox container.
- FDA *does* override the sandbox in principle, but only on processes started after the grant — and Macs running stale processes won't pick it up without a relaunch. Compounding: the path isn't TCC-protected, so users can't intuit why FDA is even relevant.

Decided to refactor to security-scoped bookmarks (proper sandbox pattern, App Store-compatible, no FDA prompt ever). User explicitly committed to keeping App Sandbox on for App Store viability — saved as durable feedback memory.

**New file:** `FolderAccessBookmarks.swift` — bookmark store keyed by Syncthing folder ID, stores in `UserDefaults` under `FolderAccessBookmark.<folderID>`, surfaces stale state to caller, has a `refresh(_:for:)` that re-saves while inside an active access scope.

## Evening pass — `/minimums` walkthrough + polish

Ran `/minimums` (the App Minimums baseline checklist). Phase 5 (polish) was already done for stuck-deletes, but the audit caught several gaps that wouldn't have surfaced from a feature-only review.

### Diagnostic logging (gap → covered)

`/minimums` flagged: app has OSLog wired across three categories (`FolderStatus`, `StuckDeletes`, `FolderAccess`), but **no file-based diagnostic log** users can attach to bug reports. Console.app + `log stream` is unfriendly for non-technical reporters.

**Approach:** OSLogStore export-on-demand rather than tee'ing every existing call site. Justification: 25 existing log sites with privacy-modifier interpolation — wrapping all of them is invasive churn. `OSLogStore(scope: .currentProcessIdentifier)` reads what's already being logged, no entitlement needed (App Store-safe; the wider `.system` scope requires `com.apple.developer.logging` which sandboxed apps can't ship with).

**New file:** `DiagnosticLogger.swift`
- `enum DiagnosticLogger` with static `export() throws -> URL`
- Writes header (export timestamp, app version, subsystem, scope) + ISO-timestamped log lines
- Format: `[ISO timestamp] [category] [level] message`
- Output path: `~/Library/Application Support/syncthingStatus/diagnostic.log` — resolves inside the sandbox container (`~/Library/Containers/com.lucesumbrarum.syncthingStatus/Data/Library/Application Support/syncthingStatus/`), which is automatic via `FileManager.applicationSupportDirectory`
- `ExportError` enum with localized descriptions for the three failure modes (store unavailable, enumeration failed, write failed)

**UI:** new "Diagnostics" section in `SettingsView` (between Notifications and Updates):
- Help text explaining the export captures the current session only
- "Export Diagnostic Log…" button → calls `DiagnosticLogger.export()`, then `NSWorkspace.shared.activateFileViewerSelecting([url])` to reveal in Finder
- Inline error rendering using the same pattern as `configSelectionError`

### Empty-export gotcha + bookend notices

First end-to-end test produced an empty export: `(no entries yet — reproduce the issue, then export again)`. Reason: by default OSLog only retains `notice`/`error`/`fault` for the archive; `info`/`debug` are kept in memory but commonly dropped from store reads on a clean session. On a working setup with no errors, **there's literally nothing to export**.

That looked like a broken pipeline even though the file structure was correct. Fix: added two `notice`-level bookend log lines that always fire:

- `appLifecycleLog.notice("App launched: \(marketing) (\(build))")` — at the top of `applicationDidFinishLaunching`. New `AppLifecycle` OSLog category in `App.swift`.
- `log.notice("Diagnostic export requested")` — first thing inside `DiagnosticLogger.export()`. New `Diagnostics` OSLog category.

Re-test confirmed: even on a clean session the file now contains both lines, proving the pipeline works. Real bugs that fire `error`-level entries get captured automatically.
